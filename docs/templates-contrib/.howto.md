# .howto

This file contains contribution templates for the major architectural areas in this repository.
Use these templates when creating or editing code so contributions remain aligned with the repo's service-first design.

See the barebones template files under this folder:

- `contracts-template.md`
- `services-template.md`
- `modules-template.md`
- `runtime-template.md`
- `external-tieins-template.md`
- `workflow-template.md`

---

## 1. Contracts

Location: `lib/contracts/`

Purpose:

- Define pure data contracts and boundary schemas.
- Keep contracts implementation-free and stable.
- Use them as the shared interface between services and consumers.

When contributing:

- Add a new contract only when a new service boundary is needed.
- Keep payload shapes explicit: name each field and document required vs optional.
- Do not put business logic or helpers in contract files.
- Prefer JSON-like shape examples and TypeScript-style type declarations if helpful.

Template:

- Contract name: `lib/contracts/<feature>.js`
- Exports:
  - `ingest<Feature>Command` (producer payload schema)
  - `emit<Feature>Event` (producer output schema)
  - optional helpers for validation or normalization

Example:

```js
// lib/contracts/party.js
module.exports = {
  PartyCommand: {
    name: "party.join",
    schema: {
      characterId: "string",
      partyId: "string",
    },
  },
  PartyEvent: {
    name: "party.joined",
    schema: {
      partyId: "string",
      memberIds: "string[]",
    },
  },
};
```

Checklist:

- [ ] Contract file lives under `lib/contracts/`
- [ ] No implementation logic in contract module
- [ ] Input/output fields documented clearly
- [ ] Consumers and producers refer to the same contract names

## 2. Services

Location: `lib/services/<feature>/`

Purpose:

- Own feature behavior and implementation details.
- Expose a public service contract through `lib/services/<feature>/index.js`.
- Validate config and wire inputs during service setup.

When contributing:

- Create a service for new feature logic, not in `lib/modules/` or runtime entrypoints.
- Keep each service focused on one responsibility.
- Use `index.js` as the service public port.
- Import contracts from `lib/contracts/` only, never from a sibling service’s internals.
- If adding config, document defaults and allowed bounds in `lib/services/<feature>/config.js`.

Template:

- `lib/services/<feature>/index.js` exports `setup()` and `teardown()`.
- `setup(container, config)` wires listeners and validates config.
- `teardown()` removes listeners and clears timers.

Example:

```js
// lib/services/party/index.js
const { PartyCommand, PartyEvent } = require("../../contracts/party");

function setup({ eventBus, storage }, config) {
  validateConfig(config);
  eventBus.on(PartyCommand.name, (payload) => {
    // service logic
    eventBus.emit(PartyEvent.name, { partyId: payload.partyId, memberIds: [] });
  });
}

function teardown() {
  // cleanup
}

module.exports = { setup, teardown };
```

Checklist:

- [ ] Service owns feature logic and exposes a public entrypoint
- [ ] Service uses contracts for boundary definitions
- [ ] `lib/services/<feature>/index.js` is the only public import
- [ ] No direct runtime or module imports into service internals
- [ ] Setup/teardown are symmetric and idempotent

## 3. Modules

Location: `lib/modules/`

Purpose:

- Provide thin adapters or integration glue for runtime components.
- Depend on services only, never own feature decisions.
- Keep module code small and focused on wiring UI, CLI, or game hooks.

When contributing:

- Add module behavior only if it is truly a UI/runtime adapter.
- Delegate logic to `lib/services/` instead of embedding it in modules.
- Avoid service-like logic or business rules in module files.
- Preserve the module as a consumer of service outputs/events.

Template:

- `lib/modules/<feature>.js` imports service public API and binds it to runtime hooks.
- If the module uses event streams, map service events to runtime events.

Checklist:

- [ ] Module only depends on `lib/services/*` public exports
- [ ] No new business logic hidden inside a module
- [ ] Module remains a bridge, not a feature owner
- [ ] Runtime hooks are centralized here, not in services

## 4. Runtime

Location: `lib/bootstrap/index.js`, `lib/al_main.js`, root startup files

Purpose:

- Bootstrap the application and wire services/modules.
- Handle environment detection, initialization order, and top-level lifecycle.
- Keep runtime thin and focused on orchestration.

When contributing:

- Add new runtime behavior only when it is environment/bootstrap specific.
- Do not implement feature business logic here.
- Use runtime to instantiate service containers and pass config.
- Validate that root startup remains safe for Node/Electron and preserves `main()` export.

Template:

- `main()` loads config, initializes services, and returns promises for startup completion.
- `lib/bootstrap/index.js` imports `lib/services/*` and registers them.

Checklist:

- [ ] Runtime only wires services/modules
- [ ] No feature logic in bootstrap code
- [ ] Root entrypoints remain stable and loader-friendly
- [ ] `main()` remains exported for external bootstrap callers

## 5. External tie-ins

Location: `local-code-server/`, `telegram/`, `agentic/`, or other integration folders

Purpose:

- Implement integrations with external tools, services, or deployment helpers.
- Keep these separate from core game and bot logic.

When contributing:

- Add external integration code only inside the relevant integration folder.
- Treat external tie-ins as consumers of core services or separate utility scripts.
- Do not mix integration config with core runtime config unless necessary.
- Keep secrets and keys out of repo; use external config or secrets files.

Template:

- `local-code-server/` for local code-server integration
- `telegram/` for Telegram bot hooks and webhooks
- `agentic/` for migration and tooling scripts

Checklist:

- [ ] External integration code is isolated by folder
- [ ] Core services/modules are not duplicated inside tie-ins
- [ ] Secrets are never committed
- [ ] Integration code documents expected environment variables/config

---

## 6. General contribution workflow

1. Identify which area the change belongs to: contracts, services, modules, runtime, or external tie-in.
2. Add or update the target file under the correct folder.
3. Keep the implementation in the area owner and use public boundaries between areas.
4. Run repository validation checks if available.
5. Update this file or the relevant documentation if a new boundary contract or service is added.

Use this file as a checklist when submitting PRs. The goal is to preserve a clear service-first architecture and avoid hidden dependencies between runtime, modules, and services.
